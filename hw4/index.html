Overview

For this assignment, we implemented a real-time cloth simulator using a mass and spring based system. We first built a grid of point masses and springs to represent the underlying structure of the cloth. We applied physical constraints to this grid, then used Verlet integration in order to simulate the motion of the cloth over time. Then, we implemented features such as collisions with other objects and self-collisions to prevent clipping. We finally implemented various types of shaders including Blinn-Phong shading, texture mapping, bump and displacement mapping, and environment mapping. Our final product realistically simulates the behavior of a sheet of cloth in real-time.

Part I

In this task, we constructed a grid of point masses and springs to represent a sheet of cloth. To do so, we created a mesh with num_width_points times num_height_points point masses evenly spanned across the cloth’s width and height. If the orientation of the cloth was HORIZONTAL, we set the y-coordinate of each point mass to 1 while varying positions over the xz-plane. If its orientation was VERTICAL, we set the z-coordinate of each point mass to a small random offset between -1/1000 and 1/1000—calculated by scaling rand()—while varying positions over the xy-plane. We used emplace_back to store these point masses in the cloth’s point_masses, in row-major order. If the (x, y) index of a point mass was stored within the cloth’s pinned vector, we set that point mass’ pinned to true. Finally, we iterated through our grid to apply structural, shearing, and bending constraints between the appropriate point masses, using emplace_back to populate the cloth’s springs.

Part II

First, we iterated over each point mass, applying its total external force. We computed the external force on each point mass using Newton’s 2nd Law:

F = ma

Where a is an acceleration from external_accelerations and m is the mass. We then iterated over each spring, applying spring correction forces to the point masses at each end. If the spring’s constraint type was enabled, we used Hooke’s Law to calculate the correction force:

Fs = ks * (||pa - pb|| - l)

ks is the spring constant, pa and pb are the positions of the two point masses, and l is the spring’s rest_length; we also multiplied the ks of our bending constraints by 0.2 to make them weaker than structural and shearing constraints. We applied the correction force to the point masses in opposite directions. Given all of the forces acting on each point mass, we performed Verlet integration to calculate a point mass’ resulting position. The new position of a point mass at time t + dt can be computed as follows, where d is a damping term:

xt + dt = xt + (l - d) * (xt - xt - dt) + at * dt2

We iterated over each point masses again. If the point mass wasn’t pinned, we updated its position, storing its previous position in last_position. Finally, we ensured that the length of each spring was at most 10% greater than its rest_length at the end of each time step by adjusting the positions of point masses as needed. If both point masses were unpinned, we performed half of the correction to both; if one was unpinned, it received the entire correction.

Part III

Next, we gave the cloth the ability to collide with other objects in the scene, namely spheres and planes. Given a point mass, we checked if its position intersected with or lay inside the sphere—if so, we corrected its position to lie on the surface of the sphere. We did this by computing the point where the point mass would’ve intersected the sphere: the tangent point. We then created a correction vector that, when applied to the point mass’ last_position, bumped it to the tangent point. We updated the point mass’ position to its last_position scaled by the correction vector and 1 - f, where f is the force of friction. We implemented collisions with planes in a similar fashion—if a point mass intersected with or crossed the plane, we bumped it to the surface of the plane it originated from, SURFACE_OFFSET above the tangent point. We updated the point mass’ positions as we did above. To apply sphere and plane collisions, we iterated over each point mass and checked for any collisions with other primitives.

Part IV

To make self-collisions more efficient, we first implemented spatial hashing. We imagined the scene as a spatial map split into 3D box volumes, each containing their own point masses. We gave each 3D volume a dimension of  w * h * t, where w = 3 * width / num_width_points, h = 3 * height / num_height_points, and t = max(w, h). Next, we assigned each point mass to a 3D volume based on a unique hash key, calculated using the following formula:

key = floor(pos[0]/w) + 10000 * floor(pos[1]/h) + 1000000000 * floor(pos[2]/t)

We iterated over the point masses and populated the spatial map with the keys calculated above. Given a point mass, we used the spatial map to find candidates for collision; we created correction vectors to set each pair of point masses 2 * thickness apart. The point mass’ final correction vector was the average of all the pairwise correction vectors, scaled by simulation_steps. Finally, we updated our implementation to check point masses not only for collisions with other primitives, but for self-collisions as well.

Explain in your own words what is a shader program and how vertex and fragment shaders work together to create lighting and material effects.

In homework 3, we implemented raytracing, which runs quite intensively on the CPU. Shaders, on the other hand, are isolated programs that run in parallel on the GPU. Shaders are written in GLSL, a language similar to C, and operate in parallel with other parts of the graphics pipeline. There are two fundamental types of shaders: vertex and fragment shaders. Vertex shaders modify geometric properties of vertices, like normals or positions. Their outputs are processed by fragment shaders, which use the geometric properties of a fragment to compute an output color vector.

The Blinn-Phong shading model combines ambient lighting, diffuse lighting, and specular reflection, and can be calculated using the following formula:

L = kaIa + kd(I / r2)max(0, n * l) + ks(I / r2)max(0, n * h)p

kaIa represents the ambient lighting, kd(I / r2) max(0, n * l) represents the diffuse lighting, and ks(I / r2)max(0, n * h)p represents the specular reflection. The following images were rendered with p = 30, ka = 0.1, Ia = 1.0, ks = 0.6, and kd = 1.0:

Explain what you did in your custom shader, if you made one.

We decided to implement a cartoon-style cell shader. In cell shading, flat colors are used to represent shadows and highlights instead of gradients. Given a point, we first calculated its Lambertian shading value. We split the output color into three ranges: a midtone color, a shadow color, and a highlight color. Depending on the range the shading value fell into, we assigned the output color to the midtone color, the shadow color, or the highlight color. This gave us the clean, solid borders we observe in cell shading.
